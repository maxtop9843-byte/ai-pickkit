# Pickkit Canonical Design Guide

이 문서는 Pickkit의 화면을 추가하거나 수정할 때 사용하는 저장소 내부의 최우선 디자인 기준이다. 라이브러리의 `DESIGN-apple.md`와 `DESIGN-mintlify.md`에서 Pickkit에 적합한 원칙을 추려 통합했다.

## 1. 제품 인상

Pickkit은 AI 비용과 모델 선택을 빠르게 판단하는 도구다. 화면은 마케팅 장식보다 입력, 비교, 결과, 근거를 우선한다.

- 전문적이되 차갑지 않다.
- 정보 밀도는 높지만 읽기 흐름은 명확하다.
- 새 기능은 기존 화면과 같은 제품에서 나온 것처럼 보여야 한다.
- 기능마다 별도의 시각 언어를 만들지 않는다.
- 장식보다 계산 결과, 비교 근거, 출처, 다음 행동을 강조한다.

## 2. 피해야 할 표현

- 흔한 보라색 또는 무지개 그라데이션
- 의미 없는 대형 영웅 문구
- 모든 요소를 둥근 카드로 감싸는 구성
- 불필요한 유리 효과와 과도한 그림자
- 장식용 3D 오브젝트와 제품과 무관한 일러스트
- 복제형 AI SaaS 대시보드 구성
- 정보 계층 없이 동일한 강도로 반복되는 카드
- 모바일에서 표와 입력이 잘리는 구성

## 3. 시각 언어

Apple의 절제와 Mintlify의 개발자 도구 밀도를 결합하되 그대로 복제하지 않는다.

### 색상

- 기본 배경은 흰색 또는 매우 옅은 중성색을 사용한다.
- 본문과 제목은 순수 검정보다 부드러운 짙은 잉크 계열을 우선한다.
- 상호작용 강조색은 한 계열만 일관되게 사용한다.
- 성공, 경고, 오류 색은 의미가 있을 때만 사용한다.
- 색상 변화 자체가 구획을 만들 수 있으면 그림자를 추가하지 않는다.

### 타이포그래피

- 시스템 글꼴, Inter, Geist 계열처럼 높은 가독성의 서체를 우선한다.
- 제목은 굵기 600을 기본으로 하고, 과도한 700~900 굵기를 남용하지 않는다.
- 본문은 16~~17px, 줄높이 1.45~~1.6 범위를 기본으로 한다.
- 숫자, 가격, 토큰, 코드, 모델 ID는 필요할 때 모노스페이스를 제한적으로 사용한다.
- 제목 크기보다 정보 구조와 간격으로 위계를 만든다.

### 간격

- 4px 기반 척도를 사용하고 8px 단위 리듬을 우선한다.
- 카드 내부 기본 여백은 20~~24px, 복잡한 비교 패널은 24~~32px를 사용한다.
- 주요 섹션 간격은 64~96px 범위에서 조정한다.
- 도구 화면은 마케팅 화면보다 촘촘하게 구성하되 답답하지 않게 한다.

### 모서리와 깊이

- 기본 반경은 8~12px를 우선한다.
- pill 형태는 주요 CTA, 필터, 명확한 선택 컨트롤에만 사용한다.
- 그림자는 기본값이 아니다. 1px hairline, 표면색 변화, 간격으로 깊이를 만든다.
- 깊은 그림자는 제품 미리보기나 특별한 부유 요소에만 제한한다.

## 4. 레이아웃 원칙

- 핵심 도구는 `입력 → 계산 또는 비교 → 결과 → 근거와 출처 → 다음 행동` 순서로 읽혀야 한다.
- 첫 화면에서 사용자가 무엇을 입력하고 무엇을 얻는지 보여야 한다.
- 결과는 페이지 하단에 숨기지 말고 입력 변화와 함께 즉시 이해되도록 배치한다.
- 비교 테이블은 첫 열과 헤더의 맥락이 유지되도록 구성한다.
- 긴 설명은 도구를 밀어내지 않도록 요약과 상세를 분리한다.
- 관련 도구와 상세 가이드는 현재 작업을 방해하지 않는 위치에 연결한다.

## 5. 컴포넌트 규칙

### 버튼

- 주요 행동은 한 화면에 하나의 시각적 우선순위를 갖는다.
- 버튼 최소 높이는 데스크톱 40px, 모바일 44px다.
- hover, active, focus-visible, disabled 상태를 구현한다.
- 아이콘만 있는 버튼은 접근 가능한 이름을 제공한다.

### 입력

- 라벨, 단위, 허용 범위, 오류 상태를 명확하게 표시한다.
- 숫자 입력은 모바일 키패드와 증감 조작을 고려한다.
- 기본값과 프리셋을 제공하되 사용자가 모든 값을 수정할 수 있어야 한다.
- 계산 결과에 영향을 주는 숨은 기본값을 만들지 않는다.

### 카드와 패널

- 카드가 필요한 이유가 있을 때만 사용한다.
- 서로 관련된 입력은 하나의 패널로 묶고, 모든 문장을 별도 카드로 쪼개지 않는다.
- 강조 카드는 한 화면에 소수만 사용한다.
- 테두리, 배경색, 간격 중 최소한의 수단으로 구획한다.

### 표와 비교

- 수치 단위와 검증 날짜를 표시한다.
- 모바일에서는 가로 스크롤, 열 축약, 비교 카드 전환 중 가장 이해하기 쉬운 방식을 선택한다.
- 중요한 차이는 색상만으로 전달하지 않는다.
- 공식 출처로 이동하는 링크를 가까이에 둔다.

## 6. 반응형 기준

- 모바일 소형: 360~479px
- 모바일 대형: 480~767px
- 태블릿: 768~1023px
- 데스크톱: 1024px 이상

모바일에서는 다음을 반드시 확인한다.

- 단일 열 읽기 흐름
- 44px 이상의 터치 영역
- 표, 코드, 긴 모델명의 넘침
- 고정 요소가 결과나 입력을 가리지 않는지
- 입력 키보드가 열렸을 때 핵심 컨트롤 접근 가능 여부
- 가로 스크롤이 페이지 전체에 발생하지 않는지

## 7. 접근성 기준

- 키보드만으로 주요 작업을 완료할 수 있어야 한다.
- 모든 상호작용 요소에 명확한 focus-visible 상태를 제공한다.
- 본문과 인터페이스 텍스트는 충분한 대비를 확보한다.
- 오류는 색상 외에 문장과 연결 관계로 전달한다.
- 입력 라벨, 설명, 오류 메시지를 프로그램적으로 연결한다.
- 동적 결과 변화는 필요할 때 적절한 live region을 사용한다.
- prefers-reduced-motion을 존중한다.

## 8. 구현 절차

UI 변경이 포함된 작업은 다음 순서를 따른다.

1. 기존 화면과 공통 컴포넌트를 조사한다.
2. 새 컴포넌트보다 기존 컴포넌트 확장을 우선한다.
3. 정보 구조를 먼저 설계하고 장식은 마지막에 판단한다.
4. 데스크톱과 모바일에서 실제 렌더링을 확인한다.
5. 핵심 입력, 계산, 공유, 초기화, 오류 흐름을 직접 조작한다.
6. 키보드 탐색, focus, 대비, overflow를 확인한다.
7. PR 본문에 디자인 판단과 시각 검증 결과를 기록한다.
8. 시각 검증이 불가능하면 완료 처리하지 않고 한계를 명시한다.

## 9. 관련 스킬 사용

사용 가능한 환경에서는 다음 스킬과 도구를 우선 활용한다.

- shadcn/ui: 접근 가능한 UI 컴포넌트와 구성 패턴
- React best practices: 상태, 렌더링, 컴포넌트 구조 검토
- Next.js: App Router, 메타데이터, 성능, 접근성 구현
- agent-browser 또는 동등 도구: Preview와 production의 실제 화면·상호작용 검증

스킬이 없거나 사용할 수 없는 경우에도 이 문서의 기준은 완화되지 않는다. 코드, 렌더된 HTML, CSS, 반응형 규칙과 접근성 속성을 근거로 대체 검증하고 한계를 보고한다.

## 10. 완료 조건

UI 작업은 아래 항목을 모두 만족해야 완료할 수 있다.

- 기존 Pickkit 화면과 시각적으로 일관됨
- 핵심 기능이 첫 시도에 이해 가능함
- 데스크톱과 모바일의 주요 흐름이 정상임
- 명백한 overflow, 잘림, 겹침이 없음
- 키보드 탐색과 focus 상태가 정상임
- 색상 대비와 오류 전달이 충분함
- 공식 데이터의 출처와 검증일이 표시됨
- Preview 또는 production에서 시각·상호작용 검증 결과가 남아 있음
